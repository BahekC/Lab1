# Задача 1. «Пересечение мультимножеств»

## Постановка
Даны два неупорядоченных списка целых чисел `A` и `B`.  
Нужно получить список `C` — пересечение мультимножеств: каждый элемент `x` входит `min(nA(x), nB(x))` раз. Порядок элементов в `C` может быть любым.

## Идея решения
1. Считаем частоты элементов одного списка в словарь `counts` (лучше — меньшего, чтобы экономить память).
2. Проходим по второму списку: если элемент есть в `counts` и его частота > 0 — добавляем в результат и уменьшаем счётчик.

## Сложность
- **Время:** `O(|A| + |B|)` (линейно по суммарному размеру)  
- **Память:** `O(min(|A|, |B|))`

## Пример
В`Task1.ipynb` приведён готовый пример входных данных и печать результата.

# Задача 2. «Интеллектуальный помощник ввода»

## Этап обучения

### 2.1. Текст → список слов
Текст читается из файла.  
Словом считается последовательность **только букв**. Разделители: пробел, перенос строки, знаки препинания `, . ; ! ? :` и другие небуквенные символы.  
Пустые слова не добавляются. Все слова приводятся к нижнему регистру. **Порядок сохраняется.**

**Пример из задания:**
`We study programming languages C++, C#, Go. We are programmers!`

Результат:
`["we","study","programming","languages","c","c","go","we","are","programmers"]`

### 2.2. Словарь биграмм
Биграмма — 2 соседних слова.  
Строим словарь:
- **ключ:** первое слово биграммы
- **значение:** список слов, которые встречались **после** этого ключа

Важно: **повторы в списке значений сохраняются**, что фактически учитывает частоту переходов.

Пример из задания (сокращённо):
- `"we": ["study", "are"]`
- `"c": ["c", "go"]`

---

## Этап автодополнения (2.3)
Пользователь вводит одно слово:
- если слово есть в словаре биграмм — выбирается случайное продолжение из списка и процесс повторяется (в сумме добавляется 3–4 слова)
- если слова нет — выводится сообщение “Слово не найдено”

Для воспроизводимости можно фиксировать seed генератора случайных чисел.

# Задача 3. Своя хеш‑функция + словарь

## 3.0. Выбор и модернизация хеш‑функции (djb2+)

За основу взята строковая хеш‑функция **djb2** (Bernstein). Используется её XOR‑вариант (**djb2a**):
`h = h * 33 ^ c`

### Улучшения (модернизация)
Чтобы повысить качество распределения на словах естественного языка:
1. **Нормализация Unicode:** `NFC` + `casefold()` (устойчивость к регистру и особенностям Unicode)
2. Хеширование по **UTF‑8 байтам**
3. 64‑битная арифметика (`& 0xFFFFFFFFFFFFFFFF`)
4. **Финальное перемешивание бит (finalizer, avalanche):** `fmix64` (константы MurmurHash3)

Идея finalizer: даже если промежуточный хеш содержит “закономерности”, финальный mixer увеличивает энтропию и снижает риск “плохого” распределения.

## 3.1. Словарь/множество на основе хеш‑функции (interactive mode)

Реализовано hash‑множество слов через **separate chaining**:
- таблица бакетов (списки)
- коллизии приводят к попаданию разных слов в один бакет

Команды:
- `add <слово>` — добавить слово
- `check <слово>` — проверить наличие (ответ `yes`/`no`)

## 3.2. Эксперимент по коллизиям и сравнение с “идеальной” функцией

### Метрика коллизий
Используется метрика “коллизии по парам” в бакетах:

Для бакета размера `k` число конфликтующих пар:
`C(k,2) = k*(k-1)/2`

Тогда общее число коллизий:
`Collisions = Σ C(k_i,2)` по всем бакетам.

### Идеальная равномерная функция
Если бы хеш‑функция распределяла `n` уникальных слов по `m` бакетам максимально равномерно, то:
- `q = n // m` (базовый размер)
- `r = n % m` (сколько бакетов будет иметь размер `q+1`)
- `r` бакетов размера `q+1`, остальные `m-r` бакетов размера `q`

Тогда:
`Ideal = r*C(q+1,2) + (m-r)*C(q,2)`

В `Task3.ipynb` выводится сравнение:
- `observed_pairs` (факт)
- `ideal_pairs` (идеал)
- `observed_pairs / ideal_pairs` (насколько хуже/лучше идеала)
